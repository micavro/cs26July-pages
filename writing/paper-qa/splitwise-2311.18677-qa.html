Splitwise：基于阶段拆分的高效生成式 LLM 推理

ISCA 2024 · 常见提问 Q1–Q7

Q1Q2Q3Q4Q5Q6Q7

Q1：论文要解决什么问题？

生成式推理的两阶段对硬件需求相反：prompt/prefill 是大矩阵计算，token generation/decode 每步 token 少、反复读权重，常受带宽限制。把两者混在同类昂贵 GPU 上会造成算力和功耗浪费。论文研究能否按阶段拆机并独立供给资源。

Q2：有哪些相关研究，区别在哪里？

早期 serving 系统主要研究 batching、并行和 KV 内存。Splitwise 与 DistServe 都属于 P/D 解耦：Splitwise 更早从硬件代际、功耗、成本和集群设计角度论证阶段专用池；DistServe 更集中于 TTFT/TPOT SLO goodput。DéjàVu 等工作提供 KV 流式传输背景。

Q3：论文如何解决这个问题？

请求先在 prompt 池完成 prefill，产生首 token 和 KV cache，再通过 NVLink/NVSwitch 等高速互连把状态传给 token 池。两个池可各自 batching 和扩容。作者构建吞吐优先、功耗优先、成本优先配置，也讨论同构和异构机器组合及少量混合节点处理流量不平衡。

\[T_{\mathrm{request}}=T_{\mathrm{prompt}}+T_{\mathrm{KV\ transfer}}+\sum_{t=1}^{N_o}T_{\mathrm{token},t}\]

拆分只有在阶段专属批处理和硬件匹配带来的收益大于 KV 传输与额外排队时才成立；因此高速 backplane 是设计前提。

Q4：论文做了哪些实验，结论可靠吗？

作者在不同 GPU 代际、模型、batch、prompt/output 长度下做 characterization，并用真实请求分布模拟集群容量、成本和能耗。代表性结果是：一种配置比共置基线吞吐高 1.4×且成本低 20%；在相同成本与功耗预算下，另一配置可达 2.35× 吞吐。

Q5：有什么局限与可进一步探索的点？

收益依赖高速互连与稳定的阶段比例。KV 很大、网络拥塞、短请求或流量突变会增加跨池代价和排队不平衡；异构节点还带来运维与容量碎片。后续可做在线弹性、KV 压缩、请求级是否拆分决策、混合池兜底，以及碳强度感知的阶段放置。

Q6：如何概括论文的主要内容？

Splitwise 的核心是按资源特征而不是按完整请求配置集群：prompt 用擅长计算的机器，token generation 用更具带宽/成本效率的机器，KV cache 作为两阶段之间的显式状态传递。这样可以分别扩容两个池并把昂贵 GPU 的算力用在真正计算密集的阶段。

Q7：系统的详细主要流程是什么？

到达请求先进入 prompt 队列并按 prompt batch 执行；完成后生成首 token、KV 与目标 token-pool 信息。传输层把 KV 经高速互连送到目标机器，token worker 注册状态并加入 generation batch，逐步输出 token。控制面持续观察两池队列，分别调整副本或把部分机器设为混合角色；容量规划器按目标函数选择同构/异构配置，并确保互连带宽能承受 KV 流。
