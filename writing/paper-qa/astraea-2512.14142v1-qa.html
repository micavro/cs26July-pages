Astraea：面向 LLM 驱动 Agent 的状态感知调度引擎：详细论文问答

Q1: 这篇论文试图解决什么问题？

Astraea 试图解决 LLM-powered Agents 的端到端 Job Completion Time（JCT）过高 问题。和普通聊天请求不同，Agent 请求通常不是一次模型调用，而是多轮 prefill/decode -> tool/API -> 再推理 的生命周期。现有系统把每个推理 segment 当作独立请求优化，无法感知下一段是否要等待 API、等待多久、KV cache 在等待期间是否还值得保留。

具体问题有三层。第一，Head-of-Line 阻塞：长计算段或长 API 等待会让短请求被拖住，局部最短或 FCFS 策略无法最小化全局 JCT。第二，资源闲置和错误保留：请求等待外部 API 时，GPU 可能没有计算该请求，但它的 KV cache 仍占内存，压缩其它请求的 batch 空间。第三，缓存策略静态：Preserve、Discard、Swap 各有代价，现有启发式通常不随内存压力、API 等待时间和重算成本动态改变。

因此论文的核心目标是把优化对象从“单个推理段的延迟/吞吐”提升到“完整 Agent 生命周期的平均 JCT”。Astraea 希望服务端理解请求状态、未来 I/O 可能性和缓存代价，在在线负载中同时做调度和 KV-cache 管理。

Q2: 有哪些相关研究？

相关研究可以分为三条线。

  • LLM 推理内存与批处理优化：PagedAttention/vLLM、LightLLM、FlashAttention、ORCA、Sarathi、DeepSpeed-FastGen、FlexGen 等工作主要改善单次推理的内存布局、连续 batching、prefill/decode 执行效率或大模型部署能力。它们很强，但默认请求是相对独立的 LLM query，不会把 API 等待和后续 segment 放入同一个调度目标。
  • 单请求或单阶段调度策略：FastServe、REEF、VTC、Splitwise、DistServe、TetriInfer、Llumnix 等研究关注抢占、公平性、prefill/decode 分离或异构请求调度。这些工作通常处理一次性请求或已知阶段，难以表示 Agent 中“计算后去等外部服务，再回来继续生成”的状态转移。
  • Agent/tool 场景的初步系统工作：Infercept 观察到 API 等待期可以 offload KV cache，但调度仍偏 FCFS；LAMPS 更关注换入/换出策略；Autellix 开始考虑 request-level LAS，但对未来 I/O 缺乏显式预测。

Astraea 的区别在于它把 计算时间预测、API 等待预测、状态感知队列调度和自适应缓存管理放在一个闭环里。它不是单纯优化 cache，也不是单纯换一个排队策略，而是把 Agent 请求作为有状态生命周期来服务。

Q3: 论文如何解决这个问题？

论文先把问题形式化为在线调度：请求由多个 segment 组成，每个 segment 可能包含模型计算和外部 API 等待；目标是最小化平均 JCT。作者证明该问题包含经典 precedence-constrained scheduling 的困难结构，因此不能期待通用多项式精确最优算法。

Astraea 的方案由三部分组成。

  • Service Time Predictor：估计每段的计算时间和 API 等待时间。Prefill 时间可由输入长度近似建模，decode token 数可由 segment 级 oracle 或历史统计估计，API 延迟则按工具类别统计。预测不追求完美，但给调度器提供“这个请求接下来会很快回来还是长时间等待”的未来信息。
  • Stateful-MLFQ 调度器：宏观上使用多级反馈队列，但队列迁移不是传统时间片，而是基于 token/计算成本和 API 让出行为。计算消耗过多的请求降级，主动进入 API 等待的请求可提高优先级，避免其回来后继续被阻塞。微观上在最高非空队列内使用增强 HRRN 分数，兼顾短任务优先和等待时间公平。
  • Adaptive KV-Cache Manager：在 GPU 内存压力低时倾向 Preserve 以降低恢复延迟；压力高时比较 Preserve、Discard、Swap 的内存-时间浪费。直觉上，如果 API 很短，保留 cache 划算；如果 API 很长且内存紧张，丢弃或换出可能更优。

这三个组件共同形成“预测 -> 调度 -> 缓存”的闭环，使系统能在不同负载下动态权衡吞吐、恢复延迟和公平性。

Q4: 论文做了哪些实验？

实验围绕“是否真的降低端到端 JCT”展开。

  • 实验环境：使用 A100 GPU 服务器和 GPT-J 6B、Vicuna-13B 等模型，构造包含多类 API/tool 等待的 Agent workload。负载强度从低到高变化，GPU 可用内存也设置多个压力档位。
  • 基线：包括 vLLM-FCFS、vLLM-SJF、vLLM-LAS、Infercept-FCFS 等，分别代表普通请求级 serving、短任务优先、LAS 思路和已有 Agent 等待期 offload 方法。
  • 核心指标：平均 JCT 是主指标，同时关注不同内存压力、不同 QPS、不同模型规模下的稳定性。论文还报告调度开销、参数敏感性和消融结果。
  • 主要发现：在最严苛的低内存/高 QPS 场景，Astraea 相比基线显著降低平均 JCT，论文报告最高约 25.5% 改善。Vicuna-13B 等更高内存压力模型上仍有稳定优势。消融显示，仅 Stateful-MLFQ 本身也能带来明显下降，说明收益不是只来自 cache 策略。调度开销在毫秒以下，占完整 JCT 的比例极小。

这些实验覆盖了模型规模、内存压力、负载强度、组件贡献、参数鲁棒性和系统开销，支撑了 Astraea 面向 Agent 生命周期优化的主张。

Q5: 有什么可以进一步探索的点？

可以继续探索的点很多。

  • 预测不确定性：当前预测主要使用统计或 oracle 信息。真实生产中 API 延迟和 decode 长度可能重尾且变化快，可引入区间预测、置信上界或在线校准，让调度对预测错误更鲁棒。
  • 更复杂的 workflow DAG：许多 Agent 并非线性 segment，而是有并行工具调用、条件分支和回溯。Astraea 可扩展到 DAG-level state-aware scheduling，与 FATE/Helium 的前瞻规划结合。
  • 更细粒度抢占：段级抢占避免频繁 KV 操作，但长 prefill 仍可能阻塞短请求。未来可结合 chunked prefill 或 block-level execution 做更细粒度调度。
  • 多目标服务质量：平均 JCT 之外，还应考虑 p95/p99、租户公平性、成本、能耗、API 费用和取消/重试行为。
  • 跨层缓存系统：Adaptive KV 管理可接入 LMCache/KVDrive 这类多层 KV 存储，形成 GPU/CPU/SSD/远端 cache 的统一决策。

总体来说，Astraea 打开了 Agent serving 的状态感知方向，但生产级系统还需要处理更复杂依赖、更不稳定外部服务和多租户约束。

Q6: 总结一下论文的主要内容。

一句话总结：Astraea 把 LLM Agent serving 从单段优化推进到全生命周期优化，用未来时间预测、状态感知分层队列和自适应 KV-cache 策略降低端到端 JCT。

论文首先指出 Agent 请求与普通 LLM query 的根本不同：它包含模型计算和外部 API 等待的交替，局部优化 segment 会造成全局 JCT 次优。随后作者形式化该在线调度问题并说明其复杂性。系统设计上，Astraea 预测每段计算/API 时间，用 Stateful-MLFQ 结合历史状态与等待公平性排序请求，并根据内存压力动态选择保留、丢弃或换出 KV cache。实验显示它在不同模型、QPS 和内存压力下都能降低平均 JCT，且系统开销很小。

主要贡献可以概括为：提出 Agent 生命周期级调度视角；把 API 等待和 KV cache 状态纳入调度；设计低开销的状态感知 MLFQ；用实验证明在 Agent workload 下比传统 vLLM/Infercept 类策略更稳定。

研究阅读补充：如果把 Astraea 放到本仓库的主线里，它最像“Agent runtime 的单机调度内核”。它不直接解决多模型选择，也不直接解决跨机器 KV placement，但它清楚回答了一个前置问题：当 Agent 请求在 GPU 计算和外部 I/O 之间来回切换时，调度器应该记录哪些状态、预测哪些未来事件、怎样把 cache 决策和排队决策合并。读这篇时应重点关注三个可迁移抽象：segment 生命周期、I/O-aware priority boost、cache preserve/discard/swap 的成本比较。后续若要做自己的系统，可以把它作为状态机和调度目标的基础，再叠加 Helium/FATE 的 workflow DAG 视角。
