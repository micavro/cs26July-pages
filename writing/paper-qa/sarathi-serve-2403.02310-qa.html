Sarathi-Serve：用 Chunked-Prefills 驯服 LLM 推理的吞吐量—延迟权衡

OSDI 2024 · 常见提问 Q1–Q7

Q1Q2Q3Q4Q5Q6Q7

Q1：论文要解决什么问题？

连续 batching 中，长 prefill 与短 decode iteration 交错。decode 优先能保护 TBT 却延迟新请求的 TTFT；prefill 优先提升吞吐却让已有流停顿。论文要找到既不让 decode 停顿、又能持续接纳新请求的单池调度。

Q2：有哪些相关研究，区别在哪里？

Orca/vLLM 提供迭代级 batching，FasterTransformer 等采用静态或请求级 batch。Sarathi-Serve 延续早期 Sarathi 的 chunked prefill，并把它系统化为 SLO-aware serving。与 DistServe/Splitwise 的物理解耦不同，它保留共置 GPU，通过切块减少干扰，因此没有跨机 KV 传输。

Q3：论文如何解决这个问题？

系统把 prompt 分成固定 token budget 的 chunks。每轮先加入所有 decode token，再用剩余预算填充 prefill chunk，形成 decode-maximal batch。长 prompt 跨多轮推进，任何一轮都不会出现巨型 prefill 阻塞。相近计算量的 batch 还能降低 pipeline parallelism 的气泡。

\[B_{\mathrm{token}}=B_{\mathrm{decode}}+\min\!\left(C,\ B_{\max}-B_{\mathrm{decode}}\right)\]

每轮先容纳所有可运行 decode，再用剩余 token budget 填入至多一个或若干 prefill chunk。chunk 上限 C 控制单轮执行时间，从而限制 token 间停顿。

Q4：论文做了哪些实验，结论可靠吗？

作者使用 Mistral-7B、Yi-34B、Falcon-180B 等模型，在单 GPU、张量并行和流水线并行配置下，按尾 TTFT/TBT 约束测最大承载率。相对 vLLM，单 A100 的 Mistral-7B 容量最高 2.6×，双 A100 的 Yi-34B 最高 3.7×，Falcon-180B 流水线配置最高 5.6×。

Q5：有什么局限与可进一步探索的点？

固定 chunk size 难以同时适配所有模型、prompt 和负载；切块也会增加调度与 kernel 次数，极短 prompt 收益有限。可扩展为在线自适应 chunk、与 P/D 解耦混合、按租户 SLO 分配 token budget，以及结合 prefix cache 避免对共享前缀重复切块。

Q6：如何概括论文的主要内容？

Sarathi-Serve 用时间上的细粒度拆分代替空间上的 P/D 拆机：把长 prefill 切成可控小块，每轮优先保护 decode，然后用剩余预算推进 prefill。这样同时限制 TBT 抖动、避免 prefill 饥饿，并通过均匀 batch 改善 GPU 与流水线利用率。

Q7：系统的详细主要流程是什么？

请求到达后 prompt 被登记为多个 chunk；调度周期收集所有活跃请求的下一枚 decode token，先组成 decode 核心。若 token budget 尚有余量，就从等待队列取 prefill chunk 填入。执行完成后更新 KV 和 prompt 进度：未完成 prompt 的请求回到 prefill 队列，刚完成 prompt 的请求转入 decode，完成输出的请求释放 KV。循环中通过 admission control 与 chunk size 保证尾 TTFT/TBT。
