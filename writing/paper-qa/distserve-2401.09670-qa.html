DistServe：解耦 Prefill 与 Decoding 的 Goodput 优化 LLM 服务

OSDI 2024 · 常见提问 Q1–Q7

Q1Q2Q3Q4Q5Q6Q7

Q1：论文要解决什么问题？

LLM 请求的 prefill 决定 TTFT，decode 决定 TPOT。共置系统把长 prefill 插入 decode batch，会让正在流式输出的请求停顿；同时两个阶段被迫共享同一张量/流水线并行方案。DistServe 要在 TTFT 与 TPOT 双重约束下最大化合规请求率。

Q2：有哪些相关研究，区别在哪里？

vLLM/Orca 主要在共置执行中改进批处理与内存，Splitwise 也做阶段拆分但更强调异构硬件、成本和功耗。DistServe 的独特处是把 SLO-aware goodput 作为一等目标，并联合搜索阶段资源、并行方式与拓扑放置。Sarathi-Serve 则选择不拆机器，用 chunked prefill 减少干扰。

Q3：论文如何解决这个问题？

系统建立独立 prefill 与 decode 实例。prefill 完成后，把该请求的 KV cache 经高速互连传到 decode 实例。规划器先 profiling 各阶段在不同并行度和 batch 下的延迟/吞吐，再找满足阶段 SLO 的单实例配置，最后确定副本数与放置；跨节点带宽较弱时优先把成对实例放在同节点。

\[G=\max_{\lambda}\left\{\lambda\;\middle|\;\Pr(T_{\mathrm{TTFT}}\le S_p)\ge 0.9,\ \Pr(T_{\mathrm{TPOT}}\le S_d)\ge 0.9\right\}\]

Goodput 不是裸吞吐：只有同时满足首 token 和逐 token 延迟目标的请求才算有效。DistServe 优化的是每张 GPU 能承载的合规到达率。

Q4：论文做了哪些实验，结论可靠吗？

评估覆盖多种模型、应用长度分布、TTFT/TPOT 目标与集群规模，并与共置式 serving 比较。DistServe 在 90% 请求满足双 SLO 的条件下，最高可承载 7.4× 请求，或在相同负载提供 12.6× 更严格 SLO。消融显示，消除阶段干扰和解除并行策略耦合都贡献收益。

Q5：有什么局限与可进一步探索的点？

解耦需要额外 GPU 池并传输完整 KV，低速网络、短 prompt 或低负载下未必划算；静态 profiling 也可能跟不上模型、硬件和流量漂移。可进一步研究弹性 P/D 比例、在线重规划、KV 压缩/增量传输、故障恢复，以及跨租户共享 decode 池的隔离。

Q6：如何概括论文的主要内容？

DistServe 把一个 LLM 请求视为两个资源特征完全不同的阶段，并用物理解耦消除互相干扰。它分别为 prefill 和 decode 选择最合适的并行度、batch 与副本数，再通过拓扑感知 KV 传输连接两池，最终优化的是双 SLO 约束下的 goodput。

Q7：系统的详细主要流程是什么？

离线阶段 profiling 模型在候选并行配置上的 prefill/decode 曲线；规划器依据目标 TTFT/TPOT 选择每池配置和副本数。在线请求先路由到负载合适的 prefill 实例，完成 prompt 计算并产生 KV；控制器选择 decode 实例并发起 KV 传输。decode 端接收状态后加入 continuous batch，逐 token 生成。监控器记录队列与 SLO，容量不足时扩展相应阶段，而无需同步扩展另一阶段。
