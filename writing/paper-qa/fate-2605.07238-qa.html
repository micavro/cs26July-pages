FATE：面向异构 LLM 工作流的未来状态感知调度：详细论文问答

  • 返回中文翻译
  • 打开 HTML 问答页

Q1: 这篇论文试图解决什么问题？

FATE 试图解决 异构 LLM workflow 调度中的局部最优破坏未来状态问题。在多阶段 workflow 中，当前 stage 放到哪个设备、用哪个模型、是否保留输出和 prefix，会影响后续 stage 的 model residency、parent-output locality、prefix reuse 和 device reachability。只看当前 ready stage 的最短延迟，可能让后续失去 cache 命中或模型驻留优势，最终端到端更慢。

因此 FATE 的目标是让调度器具备有限前瞻能力：当前选择不仅要快，还要为未来阶段保留有价值状态。

Q2: 有哪些相关研究？

相关研究有几类。

  • 经典 DAG scheduling：HEFT、CPOP 等已能处理静态异构 DAG，但通常假设计算/通信成本已知，不关心 LLM 状态复用。
  • LLM workflow serving：Parrot 暴露应用数据流，Helium 做 cache-aware workflow planning，HEXAGENT 做在线 agent DAG 调度。
  • KV/prefix reuse 与模型驻留：LMCache、EPIC、CacheBlend 等说明状态复用会显著影响推理成本。
  • 优化式调度：CP-SAT、MILP、frontier planning 等能表达复杂约束，但在线使用要控制求解开销。

FATE 的不同点在于把 future-state value 纳入调度信号，不只优化当前队列。

Q3: 论文如何解决这个问题？

FATE 的方法包括两个层次。

  • Frontier Planner：在当前 workflow frontier 上，用 CP-SAT 规划有限 horizon 内的候选调度。它不求完整未来最优，而是看一段可控窗口，捕捉当前决策对下游状态的影响。
  • Horizon-aware scoring：为候选动作打分时，不只计算当前执行时间，还考虑模型是否已驻留、父输出是否在本地、prefix 是否可复用、未来设备是否可达等信号。
  • 执行层调度：系统在在线事件到来时更新 frontier，结合 planner 输出和启发式规则选择 placement。

核心思想是：LLM workflow 中“状态”也是资源。调度器应当像保护计算资源一样保护将来可复用的状态。

Q4: 论文做了哪些实验？

实验评估 FATE 是否比局部调度更能降低端到端延迟。

  • 工作负载：包含不同结构的 LLM workflow DAG，涉及多模型、多阶段和状态复用机会。
  • 基线：局部最短、只看 placement cost、只看 reuse signal、以及若干 adapted scheduling baseline。
  • 指标：端到端 JCT、prefix reuse、model residency 命中、不同调度信号消融、solver overhead、扰动/冲突压力下表现。
  • 结果：FATE 在多种场景下比局部贪心策略更稳定，尤其当 workflow 中存在可复用 prefix 或模型驻留冲突时，future-state-aware planning 能避免早期错误放置导致的后续高成本。

实验还展示求解开销可控，说明有限 horizon planning 有在线使用的可能。

Q5: 有什么可以进一步探索的点？

进一步探索包括：

  • 更强预测模型：未来 stage 是否会执行、输出长度多大、prefix 是否可复用都可能不确定，可引入概率化 planner。
  • 与资源优化结合：FATE 决定任务放置，Coral 决定集群资源组合；二者可合并成跨层调度。
  • 动态分支与工具调用：真实 Agent workflow 往往在线展开，planner 需要处理未知子图和外部 API。
  • 多目标约束：除了 JCT，还应考虑成本、SLO、租户公平和 cache memory pressure。
  • 求解器退化策略：当 CP-SAT 超时或 workload 突发时，需要可靠的 fallback heuristic。

FATE 的价值在于提出了 future-state 视角，后续可把该视角扩展为概率化、分布式的 workflow runtime。

Q6: 总结一下论文的主要内容。

一句话总结：FATE 让 LLM workflow 调度器向前看，避免当前 placement 破坏未来可复用状态。

论文指出 LLM workflow 的执行状态包括模型驻留、父输出位置和 prefix cache，这些状态会跨 stage 影响成本。FATE 用有限 horizon 的 CP-SAT frontier planner 和状态感知打分，联合考虑当前延迟与未来复用价值。实验显示它比局部策略更能降低端到端延迟。

主要贡献是把“未来状态价值”明确引入异构 LLM workflow scheduling。

研究阅读补充：FATE 是本仓库最接近“论文选题雏形”的一篇，因为它把未来状态价值放进调度目标。读它时不要只看 CP-SAT，而要看它选择了哪些 state signals：model residency、parent-output locality、prefix reuse、future device reachability。这些信号本质上是在回答“当前动作会不会破坏后续更大的收益”。如果要继续做研究，可以把 FATE 的确定性 horizon 扩展成概率 horizon，引入分支概率、输出长度分布和 cache eviction 风险，从而和 probabilistic DAG 系列结合。
