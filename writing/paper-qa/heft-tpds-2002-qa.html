HEFT：面向异构计算系统的高性能低复杂度任务调度：详细论文问答

Q1: 这篇论文试图解决什么问题？

HEFT 论文试图解决 静态 DAG 应用在异构处理器上的任务调度问题。给定一个有向无环图，节点是任务，边是数据依赖和通信成本；每个任务在不同处理器上计算时间不同。目标是在满足 precedence constraints 的前提下，把任务映射到处理器并排序，使整体完成时间（makespan）尽可能短。

该问题一般是 NP-complete，精确求解代价高。论文的目标不是给出最优算法，而是设计调度质量高、时间复杂度低、适合实际异构环境的启发式算法。

Q2: 有哪些相关研究？

相关研究包括经典静态 scheduling 启发式。

  • List scheduling：为任务计算优先级，再按顺序选择处理器。HEFT 属于这一类。
  • Clustering algorithms：先把强通信相关的任务聚类，再映射到处理器。
  • Duplication-based algorithms：复制部分任务以减少通信等待。
  • Guided random search：用遗传算法、模拟退火等搜索更好 schedule。
  • 已有基线：MCP、ETF、DLS、MH、LMT 等，当时多面向同构系统或在异构系统上成本较高。

HEFT/CPOP 的贡献是面向有限数量异构处理器，在 schedule quality 和 scheduling cost 之间取得更好平衡。

Q3: 论文如何解决这个问题？

论文提出两个算法：HEFT 和 CPOP。

  • HEFT 第一阶段：task prioritizing。计算每个任务的 upward rank，即该任务平均计算成本加上从它到出口节点的最长后继路径成本。rank 高表示更可能处于关键路径，应更早调度。
  • HEFT 第二阶段：processor selection。按 rank 从高到低取任务，尝试把任务插入每个处理器已有 schedule 的空闲区间，选择 earliest finish time 最小的位置。插入式调度能利用空洞，比简单 append 更好。
  • CPOP。使用 upward rank + downward rank 识别 critical path，并把关键路径任务倾向放在使关键路径总执行时间最小的同一处理器上。

HEFT 更强调每个任务的最早完成，CPOP 更强调关键路径一致性。实验中 HEFT 通常更稳健。

Q4: 论文做了哪些实验？

实验设计很系统。

  • 随机 DAG 生成器：改变任务数量、图形结构、通信计算比（CCR）、处理器数量、异构程度等参数。
  • 真实应用图：除随机图外，还用实际应用 DAG 验证。
  • 对比算法：MCP、ETF、DLS、MH、LMT 等。
  • 指标：Schedule Length Ratio（SLR）、speedup、frequency of best results、average scheduling time。
  • 结果：HEFT 在多数参数组合下取得更低 SLR、更高 speedup 和较低调度开销。CPOP 也优于许多基线，但在通信和异构性复杂时不如 HEFT 稳定。

实验的意义在于证明 HEFT 不只是某个例子好，而是在图结构、CCR 和异构程度变化下都可靠。

Q5: 有什么可以进一步探索的点？

进一步探索方向包括：

  • 不确定执行时间：现实任务时间可能是分布而非常数，可结合 probabilistic DAG 研究。
  • 在线 DAG：Agent workflow 和云任务可能运行时才展开，不符合静态已知 DAG 假设。
  • 缓存和状态复用：LLM workflow 中任务放置会影响 KV cache 和模型驻留，HEFT 没有建模。
  • 多目标调度：除 makespan，还可考虑成本、能耗、公平性和 SLO。
  • 与现代异构硬件结合：GPU、NPU、CPU、SSD/网络传输的成本模型远比 2002 年更复杂。

HEFT 仍可作为强基线，但新问题需要在 rank 和 processor selection 中加入概率、状态和资源压力。

Q6: 总结一下论文的主要内容。

一句话总结：HEFT 是异构 DAG 静态调度中最经典的低复杂度高质量 list scheduling 基线之一。

论文把任务优先级定义为 upward rank，把处理器选择定义为 earliest finish time 插入式选择，并与 CPOP 一起在随机和真实 DAG 上评估。结果显示 HEFT 能在低调度成本下得到接近优的 schedule。它的思想简单、可解释、可扩展，因此至今仍常被用作 workflow scheduling 对照。

研究阅读补充：HEFT 是经典基线，读它的目的不是寻找最新系统技巧，而是掌握 DAG 调度语言。后续无论研究 Agent workflow、LLM serving 还是实时 DAG，都绕不开 priority rank、critical path、processor selection 和 insertion-based scheduling 这些基本概念。对本仓库而言，HEFT 可作为“没有概率、没有 cache、没有在线负载”的零阶模型；任何新算法都应能解释自己相对 HEFT 多建模了什么状态、付出了什么复杂度、换来了什么收益。

更具体地说，读 HEFT 时可以把它拆成两个可替换接口：第一是 priority function，决定 ready tasks 的先后；第二是 placement function，决定某个 task 放到哪个 processor 的哪个空隙。现代 LLM workflow 研究可以保留这两个接口，但把 upward rank 换成包含 token length、branch probability、cache reuse value 的 rank，把 earliest finish time 换成包含 queueing、model residency、KV transfer 的 finish-time estimator。这样 HEFT 就不仅是基线，也是一套可扩展的研究脚手架。
