RHRS：面向可编程 LLM 推理系统的细粒度批调度：详细论文问答

Q1: 这篇论文试图解决什么问题？

RHRS 试图解决 可编程 LLM 推理系统中的细粒度 component batch scheduling问题。传统 LLM serving 往往把一次调用看成 prefill/decode 组成的单个 request；但 workflow-centric 应用会包含多步 reasoning、tool use、agent collaboration 和动态依赖。把它们继续当成单体 request，会看不见跨步骤依赖、component type 差异和运行时状态变化。

论文的目标是在 dependency、batching 和 memory constraints 下最小化 Average Completion Time（ACT），同时避免只追求当前 batch size 造成后续 component 解锁变慢、短请求等待长请求、GPU utilization 下降等问题。

Q2: 有哪些相关研究？

相关研究可以分成几类。

  • 传统 LLM serving：Orca、vLLM 等通过 iteration-level scheduling、continuous batching 和 KV-cache management 提升单体请求吞吐。
  • Prefill/decode 优化：Sarathi-Serve、DistServe、Splitwise 等利用 prefill 与 decode 的资源差异做 chunking、phase separation 或资源解耦。
  • 可编程推理系统：Parrot、SGLang、PIE 等把 LLM 应用结构或执行组件暴露给 serving runtime，但不一定系统优化 component-level ACT。
  • DAG 与 workflow 调度：HEFT、FATE、HEXAGENT 等说明依赖结构和未来状态会影响调度，但 RHRS 关注的是 programmable inference 内部的 fine-grained batch formation。

RHRS 的切入点是把 component requests 当作在线调度实体，而不是只调度完整请求或粗粒度阶段。

Q3: 论文如何解决这个问题？

RHRS 用一个在线启发式框架，在每个 scheduling epoch 中选择一个 component-family batch。它先按 component family 划分 ready GPU-bound component requests，例如 EMBED、SAMPLE、PREFILL FORWARD、DECODE FORWARD 等；候选 batch 必须满足同类型、batch-size、token budget、memory capacity 和拓扑依赖约束。

方法分三步：

  • Candidate batch construction：用 age、batching gain、balanced、progress 四类偏好做 greedy expansion，生成多样候选。
  • Lightweight pruning：用 batching gain、平均等待时间和 progress/unlocking 信号排序，保留 top-M candidates。
  • Limited-horizon rollout：在 shadow state 中模拟短 horizon，估计每个候选对 cumulative flow time 的影响，选择估计 ACT 最低的 batch。

核心思想是让当前 batch 决策同时考虑 immediate batching efficiency、request urgency 和 downstream unlocking effects。

Q4: 论文做了哪些实验？

论文用 discrete-event simulation framework 评估不同调度策略。实验模拟 request arrivals、batch completions、component readiness、topological dependencies、batching constraints 和 memory-capacity constraints，并用 profiling 得到的 component execution-time model 估计 latency。

  • Workload：Short、Text、Agent、Mixed 四类 workload，其中 Agent 和 Mixed 更能体现多阶段依赖与动态执行路径。
  • Baselines：PIE-style queue-driven policy、MaxFill-Greedy、Sarathi-Serve adapted baseline。
  • Metrics：Average Completion Time、P95 tail latency、throughput。
  • Results：RHRS 在不同 arrival rates、workload types、request populations 和 batch-size limits 下通常取得最低 ACT；代表性 workload 上 ACT 降低 26% 到 45%，P95 tail latency 最高降低 14.8%，throughput 最高提升 22.9%。

消融实验还显示，rollout、batch-awareness 和 urgency-awareness 都有贡献，其中去掉 rollout 会显著恶化 ACT。

Q5: 有什么可以进一步探索的点？

可以继续探索几个方向。

  • 真实系统集成：论文主要在 simulator 中评估，后续需要接入实际 programmable serving runtime，处理 kernel-level latency、cache eviction 和并发干扰。
  • 多 GPU 与跨设备调度：RHRS 当前重点是 component batch selection，可扩展到 distributed multi-GPU settings，加入 communication-aware scheduling。
  • 学习式 cost model：execution-time model 可以从简单 profiling 扩展成在线校准或 learning-based predictor。
  • 动态 workflow 可见性：tool calls 和 agent decisions 会让未来 component 部分未知，rollout 可进一步结合概率分支和输出长度分布。
  • 和状态复用系统结合：可把 RHRS 的 batch-level decision 与 FATE/SAIL 的 state-aware placement、EPIC/LMCache 的 KV reuse substrate 结合。

这些方向会把 RHRS 从细粒度 batch policy 推向完整 workflow-aware serving scheduler。

Q6: 总结一下论文的主要内容。

一句话总结：RHRS 让可编程 LLM 推理系统按 component 级别做批调度，并用短期前瞻避免局部 batch 贪心损害整体 ACT。

论文指出，workflow-centric LLM 应用不能再简单看成单体 request。细粒度 components 的 type、token length、dependency、memory footprint 和 unlocking value 都会影响调度质量。RHRS 通过候选 batch 构造、轻量 pruning 和 limited-horizon rollout，在 immediate batching efficiency 与 downstream progress 之间折中。

研究阅读补充：RHRS 对本仓库的意义在于补上了“可编程推理系统内部怎么调度 component batch”的一层。它和 SAIL/FATE 的关系很清楚：SAIL/FATE 更偏 workflow-level placement 和状态价值，RHRS 更偏 runtime 内部 ready components 的批形成。如果后续要做端到端 agent serving，可以把 RHRS 作为执行层 scheduler，把 SAIL/FATE 作为更上层的 workflow placement 或 state-affinity planner。
