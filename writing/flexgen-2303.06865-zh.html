FlexGen：单 GPU 上的高吞吐大语言模型生成式推理：中文精读与校订

Ying Sheng, Lianmin Zheng, Binhang Yuan, Zhuohan Li, Max Ryabinin, Daniel Y. Fu, Zhiqiang Xie, Beidi Chen, Clark Barrett, Joseph E. Gonzalez, Percy Liang, Christopher Re, Ion Stoica, Ce Zhang
源文件：arXiv 2303.06865 | ICML 2023 | 23 页 | 保留术语：offloading、throughput-oriented inference、KV cache、weight compression、group-wise quantization、linear programming

这篇论文做什么？

面向延迟不敏感的离线生成，把权重、激活与 KV cache 统一放到 GPU/CPU/磁盘三级存储，以线性规划搜索卸载比例和 batch 参数，并用 4-bit 压缩扩大可执行批量。

  • 问题：超大模型单卡放不下；朴素逐层卸载被 PCIe/磁盘 I/O 卡住。
  • 方法：统一张量放置、块状调度、I/O—计算重叠、LP 搜索与 4-bit 压缩。
  • 结果：OPT-175B 可在单张 16GB GPU 上以有效 batch 144 达到约 1 token/s。

校订说明：本页按原论文的论证顺序重写术语与因果关系；图均裁自论文原图，数据结论保留论文适用条件，不把峰值结果泛化为固定加速。

摘要。大语言模型推理的高计算和内存需求使其只能在多个高端加速器上运行。受延迟不敏感的批量处理任务需求驱动，论文开创了使用有限资源（如单张商品级 GPU）进行高吞吐 LLM 推理的研究。FlexGen 是一个高吞吐生成引擎，可以在有限 GPU 内存下运行 LLM。FlexGen 可以在各种硬件资源约束下灵活配置，通过聚合 GPU、CPU 和磁盘的内存与计算。通过求解线性规划问题，搜索高效的张量存储和访问模式。FlexGen 进一步将权重和注意力缓存压缩到 4 bit，精度损失可忽略。这些技术使 FlexGen 拥有更大的 batch size 选择空间，从而显著提升最大吞吐量。在单张 16GB GPU 上运行 OPT-175B 时，FlexGen 相比 SOTA 卸载系统实现显著更高吞吐量，首次达到 1 token/s 的生成吞吐量（有效 batch size 144）。

1 引言

除了交互式应用（如聊天机器人），LLM 还广泛用于"后台"任务：基准测试、信息抽取、数据整理、表单处理。这些任务需要对大量 token 批量运行 LLM 推理，但对延迟不敏感。因此可以用延迟换吞吐量，降低资源需求。论文称这一场景为吞吐量导向的生成式推理。

在单张商品级 GPU（如 NVIDIA T4 16GB）上运行 175B 模型面临极端挑战：模型权重需 325GB，远超 GPU 内存。现有方案有三类：模型压缩、协作推理、卸载（offloading）。现有卸载系统（DeepSpeed Zero-Inference、HuggingFace Accelerate）因低效的 I/O 调度和张量放置，在单 GPU 上吞吐量极低——batch size 仅 1-2。

2 挑战

2.1 卸载策略设计

生成式推理中有三类张量：权重（weights）、激活值（activations）和 KV cache。卸载策略需指定：卸载哪些张量、在 GPU-CPU-磁盘三级内存层级中放在哪里、推理过程中何时卸载。批-批、token-token、层-层的计算结构形成复杂依赖图，存在多种执行方式。现有系统从训练继承策略，对推理而言次优，执行过多 I/O，吞吐量远低于硬件理论上限。

2.2 压缩策略

将压缩与卸载结合用于高吞吐推理时，权重和 KV cache 的 I/O 成本和内存减少变得更为重要，需要替代性压缩方案。

3 FlexGen 设计

3.1 卸载搜索空间

FlexGen 形式化定义了卸载策略的搜索空间，考虑计算调度、张量放置和计算委托三个维度。论文证明搜索空间覆盖的计算顺序的 I/O 复杂度在最优解的 2 倍以内。然后开发基于线性规划的搜索算法，在搜索空间内优化吞吐量。算法可配置不同硬件规格，可扩展纳入延迟和吞吐量约束。

3.2 统一张量放置

与传统系统将权重、激活值和 KV cache 分开管理不同，FlexGen 统一三者的放置策略。这使得 batch size 的上限大幅提高——权重可以常驻 CPU 或磁盘，仅在计算对应层时按需加载到 GPU，而 KV cache 可以根据可用内存灵活分布在 GPU/CPU/磁盘之间。大的 batch size 是实现高吞吐的关键：I/O 开销被大批量输入分摊。

3.3 压缩

FlexGen 展示了可以将 OPT-175B 等模型的权重和 KV cache 压缩到 4 bit 而无需重训练或校准，精度损失可忽略。这通过细粒度 group-wise 量化实现：将权重矩阵按行分组，每组独立量化。这种量化方式适合减少卸载场景下的 I/O 成本和内存使用。4-bit 压缩使所有权重可放入 CPU 内存，免除磁盘卸载的需要。

4 关键洞察

生成式推理的计算结构有一个关键特性：虽然每步只生成一个 token，但同一个模型的同一层在处理不同请求时共享权重。因此，将多个请求的同一层计算批量处理，可以用一次权重加载服务多个请求——这就是大 batch size 分摊 I/O 的原理。FlexGen 的线性规划优化器正是利用这一结构，寻找最优的权重加载和 KV cache 访问模式。

另一个洞察是权重的访问模式是确定性的：推理过程中各层权重的访问顺序固定，可以精确预取和流水线化。而 KV cache 的访问取决于生成长度，具有不确定性，但可以通过合理的放置策略（GPU 存最近 token 的 cache，CPU/磁盘存较早的）来管理。

5 实验评估

在 NVIDIA T4（16GB）GPU、208GB CPU DRAM 和 1.5TB SSD 上评估 OPT-175B 和 OPT-30B。

主要结果：在相同延迟 5000 秒下，FlexGen（有效 batch size 64）比 DeepSpeed Zero-Inference（batch size 1）高 40 倍吞吐量，HuggingFace Accelerate 无法完成单个 batch。允许 12000 秒延迟时，FlexGen 达到 69 倍最大吞吐量（有效 batch size 256）。启用 4-bit 压缩时，FlexGen 达到 100 倍最大吞吐量（有效 batch size 144），延迟 4000 秒——通过将所有权重放入 CPU 免除磁盘卸载。

FlexGen 还与去中心化协作推理系统 Petals 对比，在每 GPU 吞吐量上优于 Petals 集群，在特定配置下甚至延迟更低。

6 局限与选题启发

FlexGen 面向吞吐量导向的离线场景，延迟在分钟到小时级别，不适合交互式服务。其卸载策略虽然是线性规划优化的，但受限于三级内存层级的物理带宽——GPU-CPU 间 PCIe 带宽和 CPU-磁盘间带宽是硬瓶颈。此外，4-bit 量化虽然精度损失小，但对某些模型和任务可能不够鲁棒。

对本仓库方向的意义。FlexGen 是 LLM 推理资源受限场景的先驱工作。它揭示了通过卸载和压缩在商品硬件上运行超大模型的可能性，以及 I/O 感知的张量放置策略的重要性。在 agentic workflow 场景下，当 GPU 资源不足以同时服务所有 agent 时，FlexGen 的卸载思想可以作为后备策略——将不活跃 agent 的 KV cache 卸载到 CPU/磁盘，为活跃 agent 释放 GPU 内存。这与 vLLM 的抢占式换出策略一脉相承，但 FlexGen 更深入地优化了卸载的 I/O 调度。

公式与目标函数

\[\max_{b,g,p}\ \operatorname{Throughput}(b,g,p)\quad\text{s.t. }M_{GPU},M_{CPU},M_{disk}\text{ and bandwidth constraints}\]

b、g、p 可理解为 batch、GPU batch 分块和各类张量放置比例。LP 用硬件带宽/容量与算子代价估计稳态瓶颈，选择能最大化离线吞吐的执行策略。

证据边界与校订结论

FlexGen 面向 throughput-oriented、latency-insensitive workload，不能把其 1 token/s 解读为交互式体验。论文的重要贡献是让极小 GPU 能运行此前不可运行的模型，并通过大 batch 摊薄权重搬运；与 vLLM 这类在线低延迟服务目标不同。
